## Problemex — 임계영역 보호 (메시지 큐에 동시에 쓰기 등)
### Thread 핸들링  — 클라이언트당 처리 스레드 분리, 메시지 브로드캐스트는 락 안에서 처리
  네트워크 프로그래밍 수업의 텀 프로젝트. \
  요구사항: "채팅 서버를 만들어라". 보통은 적당한 라이브러리 가져다 빠르게
  끝내는데, 그러면 정작 소켓이 어떻게 움직이는지, 동시성 충돌이 어디서 나는지 몸으로 배우는 기회를 놓친다고 판단했다.
  thread-safe인지 일일이 그려보면서 락 범위를 결정하는 작업이었다.
  라이브러리에 가려진 부분을 직접 만나보고 싶어서, 프로토콜 레이어는 가능한 한 손으로 짜는 쪽으로 잡았다.
## Insight
### Approach
  이전에 책으로 봤던 "thread-safe 하게 짜라"가 막연했는데, 이 프로젝트로 **"thread-safe하지 않으면 어떻게 깨지는지"**
  직접 구현해볼 수 있었다. 메시지가 섞이거나, 클라이언트 연결 해제 시 다른 스레드가 잘못된 fd를 잡거나, 의외로 흔한
  버그를 다 마주쳐봤다.
  - 소켓 기반 다중 클라이언트 — socket(), bind(), listen(), accept() 직접
  - HTTP 정적 서빙 — 헤더/Content-Type/바디를 직접 작성, 같은 포트에서 처리대신 **"왜 그 라이브러리가 그런 형태로
  - WebSocket(RFC 6455) 전 구간 — 핸드셰이크의 키 교환부터 프레임 마스킹·가변 길이 인코딩까지 직접 구현. OpenSSL은
  SHA1/Base64에만 사용
  - JSON — 처음엔 자체 파서를 시도했지만 메시지 스키마가 늘면서 안정성/시간 트레이드오프 때문에 cJSON을 vendor 해 쓰는
  쪽으로 정리

## 동시성 모델 — 가장 시간을 많이 쓴 부분

  처음 설계는 직관적인 "클라이언트당 한 스레드 + 공유 자료구조에 mutex" 모델이었다. 그런데 critical section을 종이에
  그려가며 락 범위를 잡다 보니 두 가지 패턴이 매번 깨졌다.

  - 메시지 브로드캐스트 도중 한 클라이언트가 disconnect → 다른 스레드가 이미 닫힌 fd를 잡고 send()
  - 클라이언트 리스트 순회 중 다른 스레드가 노드 제거 → use-after-free

  큰 락으로 묶으면 동시성이 사라지고, 잘게 쪼개려고 하면 정합성이 깨졌다. 그래서 모델을 갈아엎고 "단일 스레드 epoll
  edge-triggered reactor + non-blocking I/O" 로 정리했다.

  - 모든 I/O 이벤트가 한 스레드에서 직렬화되므로 공유 자료구조에 락 자체가 필요 없다
  - 연결 종료(epoll DEL → close → 리스트 제거 → free)를 한 시점에 묶을 수 있어 dangling fd 문제가 구조적으로 사라진다
  - WebSocket은 long-lived idle 연결이 많아 스레드-당-연결의 비용이 이득에 비해 컸다

## Insight

  이전에 책으로 봤던 "thread-safe 하게 짜라" 가 막연했는데, 이 프로젝트로 "thread-safe하지 않으면 어떻게 깨지는지" 를
  직접 경험할 수 있었다. 메시지가 섞이거나, 끊긴 fd로 send가 가거나, 사라진 노드를 다른 곳에서 참조하거나 — 의외로 흔한
  버그를 다 마주쳐봤다.

  거기서 더 본질적으로 배운 건, "락을 어디까지 걸지"보다 "공유 상태에 접근하는 경로 자체를 한 스레드로 묶을 수 있는가"가
   먼저 던져야 할 질문이라는 것. 처음엔 mutex로 풀려고 매달렸지만, 결국 동시성을 피하는 설계가 더 단순하고 안전한 경우가
   있다는 걸 손으로 깨달았다.

  라이브러리를 안 썼기 때문에 디버깅 폭이 컸다는 게 이 결정의 비용이고, 그 대신 WebSocket의 마스킹 키나 가변 길이
  페이로드 같은 "라이브러리가 감춰주던 프로토콜의 모서리"를 직접 만나면서 "왜 그 라이브러리가 그런 형태로 설계됐는지" 를
   이해하게 된 게 이득이었다.
