## Problem

시각장애인이 일상에서 인터넷을 쓸 때, 기존 스크린리더는 두 가지에서 무너진다.

1. **이미지를 읽지 못한다** — 광고 배너의 핵심 텍스트, 차트, 캡션 없는 사진은 사실상 빈 공간
2. **동적 DOM에서 오작동한다** — 광고·팝업·SPA 라우팅에서 읽는 위치가 어긋나거나 같은 영역을 반복해서 읽음

기술이 늘어났는데도 접근성은 따라가지 못하는 구조적 격차. 특히 시간이 지날수록 격차가 더 벌어지는 방향이라는 게 문제 인식의 핵심이었다.

## Approach

### 4-Layer 시스템

4개 레이어 모두 단독으로 설계·구현했다.

- **Sensory Layer** — Faster-Whisper(STT) + Melo-TTS(TTS). 음성 입출력 담당
- **Routing Layer** — L0~L2 의도 분류. 사용자 발화를 적절한 도구로 라우팅
- **Action Layer** — 5개 도구(검색, 요약, 페이지 탐색, 양식 입력, 외부 호출)
- **Browser Layer** — 두 개의 브라우저 전략을 사이트 특성에 따라 스위칭

### Browser Strategy — 핵심 트레이드오프

봇 차단 사이트가 골치였다. CDP로 분석·조작을 모두 통합하면 깔끔하지만 차단 당한다. 그렇다고 분석을 포기할 수도 없다. 결국 둘로 나눴다.

- **기본 사이트** — Chrome CDP로 분석·조작 통합 (단순한 경로)
- **봇 차단 사이트** — 분석만 CDP로, 조작은 사용자 브라우저의 접근성 트리 기반 시스템으로 우회

이 분기로 **차단 회피 + 사용자 신원 보호**를 동시에 달성. CDP는 자동화 흔적이 강해서 신원이 노출될 위험이 있는데, 조작을 유저 브라우저로 보내면 그 흔적도 사라진다.

### Vision 대신 접근성 트리

브라우저 자동화의 표준 접근법은 Vision 모델로 화면을 보는 것이다. 하지만 비교해보니:

- **비용** — Vision API 호출당 비용 vs 접근성 트리 파싱 비용
- **지연** — Vision 추론 시간 vs 트리 노드 직접 읽기
- **정확도** — Vision은 OCR + 의미 추론 누적 오차 vs 트리는 구조화된 정답

세 축 모두 접근성 트리 우위. Vision은 트리에서 누락된 케이스(이미지 캡션 등)에만 보조적으로 사용하도록 역할 분리.

## Insight

"AI로 한 번에 해결한다"는 유혹이 큰 분야였지만, 실제로 강한 시스템은 **"AI에게 안 맡길 부분을 명확히 가르는 것"** 에서 나온다는 것을 직접 체감했다. Vision 모델 채택을 거부한 결정은 처음에는 고집처럼 보였지만, 결국 비용·속도·신뢰도 모든 차원에서 정답이었다.

접근성이라는 도메인 특성상 "잘 되는 케이스"보다 "안 되는 케이스가 0인지"가 중요했고, 그게 Vision-first 접근으로는 보장되지 않았기 때문이다.
